# 车联网实时位置数据存储设计方案

## 1. 需求分析

### 1.1 数据特性
- **高写入频率**：每辆车每2秒产生一条位置数据
- **大数据量**：5000台车 → 约2.16亿条/天；50000台车 → 约21.6亿条/天
- **时间序列数据**：位置数据具有严格的时间顺序
- **地理空间特性**：包含经纬度信息，需要支持地理空间查询
- **冷热数据分离**：近期数据访问频繁，历史数据访问较少

### 1.2 业务需求
- **实时地图展示**：在地图上实时显示车辆位置
- **轨迹回放**：支持查看历史轨迹，回放车辆行驶过程
- **地理围栏查询**：支持查询特定区域内的车辆
- **路径分析**：支持计算车辆行驶距离、速度等
- **实时监控**：支持实时监控车辆状态

## 2. 存储架构设计

### 2.1 数据库选型
- **主数据库**：PostgreSQL + PostGIS（空间数据处理能力强，支持时间分区）
- **缓存层**：Redis（存储热点数据和实时位置）
- **消息队列**：Kafka（处理高并发写入，削峰填谷）

### 2.2 表结构设计

#### 2.2.1 核心表结构

| 表名 | 用途 | 关键特性 |
|------|------|----------|
| `vehicles` | 车辆基本信息 | 主表，存储车辆静态数据 |
| `vehicle_realtime_locations` | 实时位置表 | 单条记录/车，存储最新位置 |
| `logistics_tracks` | 轨迹点表 | 时间分区，存储所有历史轨迹点 |
| `logistics_track_lines` | 轨迹线表 | 预计算轨迹线，优化回放性能 |

#### 2.2.2 数据流转
1. 车辆位置数据 → Kafka消息队列 → 消费服务
2. 消费服务将数据写入：
   - `logistics_tracks`（历史轨迹）
   - `vehicle_realtime_locations`（实时位置，UPSERT操作）
   - Redis缓存（热点数据，TTL 1小时）
3. 定时任务（每5分钟）：
   - 从`logistics_tracks`生成`logistics_track_lines`
   - 清理过期缓存

## 3. 性能优化策略

### 3.1 空间数据优化
- **使用PostGIS**：将经纬度转换为`GEOMETRY(Point, 4326)`类型
- **空间索引**：为地理空间列创建GIST索引
- **预计算轨迹线**：提前计算并存储轨迹线，减少轨迹回放时的计算量

### 3.2 时间序列优化
- **时间分区**：按天或按小时对`logistics_tracks`进行分区
- **分区裁剪**：查询时自动过滤无关分区
- **冷热数据分离**：将超过30天的数据迁移到冷存储

### 3.3 写入优化
- **批量写入**：Kafka消费端批量写入数据库
- **UPSERT操作**：使用`INSERT ... ON CONFLICT DO UPDATE`更新实时位置
- **异步写入**：非关键数据采用异步写入方式

### 3.4 查询优化
- **复合索引**：创建`(vehicle_id, track_time DESC)`等复合索引
- **Redis缓存**：热点数据（如实时位置）缓存到Redis
- **预聚合**：提前计算常用统计数据（如车辆总数、在线率）

## 4. 实时位置数据处理流程

### 4.1 数据写入流程
```
车辆终端 → 网关服务 → Kafka → 轨迹处理服务 → 数据库/Redis
```

### 4.2 数据读取流程
- **实时位置查询**：Redis → 数据库（缓存未命中时）
- **轨迹回放查询**：logistics_track_lines → logistics_tracks（细节查询）
- **地理围栏查询**：PostGIS空间索引 + 过滤

## 5. 分表分库策略

### 5.1 当前阶段（5000台车）
- **数据库**：单PostgreSQL实例
- **分表策略**：
  - `logistics_tracks`：按天分区
  - `weighing_data`：按天分区
  - `audit_logs`：按天分区

### 5.2 中期阶段（50000台车）
- **数据库**：读写分离（1主2从）
- **分表策略**：
  - `logistics_tracks`：按小时分区
  - 增加`vehicle_id`哈希分区

### 5.3 长期阶段（20万+台车）
- **数据库**：水平分片
- **分片键**：`vehicle_id`（哈希分片）
- **分片数量**：根据负载动态调整
- **分布式事务**：使用最终一致性

## 6. 缓存策略

### 6.1 Redis缓存设计
- **缓存结构**：
  - `vehicle:realtime:{vehicle_id}`：存储车辆实时位置（JSON格式）
  - `fence:vehicles:{fence_id}`：存储围栏内车辆ID列表
  - `track:summary:{vehicle_id}:{date}`：存储轨迹摘要信息
- **TTL设置**：
  - 实时位置：1小时
  - 围栏信息：10分钟
  - 轨迹摘要：24小时

### 6.2 缓存更新策略
- **主动更新**：数据写入时同时更新缓存
- **被动失效**：缓存过期后自动失效
- **批量更新**：定时批量更新热点数据

## 7. 数据备份与恢复

### 7.1 备份策略
- **全量备份**：每天凌晨1点进行全量备份
- **增量备份**：每小时进行增量备份
- **WAL日志**：实时归档WAL日志，支持PITR恢复

### 7.2 恢复策略
- **常规恢复**：使用最近全量备份+增量备份恢复
- **PITR恢复**：使用WAL日志进行时间点恢复
- **异地备份**：备份数据存储到异地，防止灾难发生

## 8. 监控与维护

### 8.1 监控指标
- **写入性能**：每秒写入条数、写入延迟
- **查询性能**：查询响应时间、QPS
- **存储使用率**：磁盘空间使用情况、分区大小
- **缓存命中率**：Redis缓存命中率
- **系统负载**：CPU、内存、IO使用率

### 8.2 维护任务
- **定期清理**：清理过期数据和无用缓存
- **索引优化**：根据查询日志优化索引
- **统计信息更新**：定期更新数据库统计信息
- **性能测试**：定期进行性能测试，发现瓶颈

## 11. 结论

本设计方案基于PostgreSQL+PostGIS构建，充分利用了PostgreSQL的空间数据处理能力和时间分区特性，结合Redis缓存和Kafka消息队列，能够满足车联网实时位置数据的存储和查询需求。方案具有良好的扩展性，能够支持从5000台车到100万台的平滑演进。

通过合理的表结构设计、索引优化和缓存策略，可以有效提高系统的写入性能和查询性能，降低系统负载，为车联网业务提供可靠的数据支撑。
